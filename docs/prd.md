# 智能学伴系统 · 产品需求文档（PRD）

> 版本：v0.3 · 状态：草案（修订） · 最后更新：2026-07-11
> 变更：v0.3 修复需求审核——资料解析提至 P0 基座、班级码+审批(F2.5)、shared 仅 teacher 生效、补测评/计划/画像/token 表、订阅额度接限流、K12 内容安全、路线图重排

---

## 1. 产品概述

### 1.1 一句话定位

> 一个由多智能体驱动的「AI 学伴」，让学生既能自主按节奏学习，也能随时获得讲解、规划与测评；资料以「团队知识库」组织，越用越懂你。

### 1.2 产品愿景

让每个学习者都拥有随时在线、懂资料、记得住进度的专属学伴，并让老师能轻松运营自己的学习小组。

---

## 2. 目标用户

| 用户群 | 特征 | 核心诉求 |
|--------|------|----------|
| K12 学生 | 课业重、需要答疑与错题 | 即时讲解、作业辅助、考点梳理 |
| 大学生 | 自驱但资料散 | 资料整理、复习规划、问答 |
| 职业培训者 | 考证/技能提升 | 结构化计划、模拟测评 |
| 终身自学者 | 兴趣驱动 | 个性化路径、陪伴感 |

### 2.1 三类角色与团队（知识库）模型

系统以「团队 = 知识库」组织资料与权限，分三类角色：

- **学生**：拥有系统自动生成的**私人 team**，可上传自己的资料（仅自己可见）；可加入老师的 team，访问其中「对学生公开」的资料；默认可访问公共库。
- **老师**：可创建**学习小组 team**（类似班级/社群），**仅老师能上传资料**，并逐份设置该资料是否对 team 内学生可见（草稿 vs 公开）。
- **超级管理员**：维护系统级**公共库**，上传的资料全平台可见，作为通用知识底座。

> 每个 team 都是一个独立知识库：资料上传后由 Agent 结构化解析入库，RAG 检索严格限定在用户有权访问的 team 集合内。

---

## 3. 用户痛点（为什么做）

1. **学不进去**：缺乏节奏与计划。
2. **没人答疑**：遇到问题卡住，等不到人讲。
3. **资料碎片化**：资料散落，检索难、不成体系——团队知识库解决「资料归属与复用」。
4. **反馈缺失**：学了多久、掌握多少全凭感觉。
5. **一刀切内容**：不同水平需要不同讲法。

---

## 4. 核心价值主张

- **双模式**：自主学习 + AI 辅助自由切换。
- **团队知识库**：老师建组、学生私有、公共库，资料各归其位、权限清晰。
- **基于资料的真懂**：RAG + 引用溯源，不是瞎编。
- **个性化**：记得住进度与薄弱点。
- **闭环**：学 → 问 → 测 → 反馈 → 再学。

## 5. 功能需求

## 6. 用户故事与典型流程

**故事 1（答疑）**
> 作为一名高中生，当我卡在一道物理题时，我希望选中题目直接问 AI 学伴，以便立刻得到基于课本的讲解和推导。

流程：打开资料 → 选中题目 → 点「问学伴」→ 流式回答（附引用）→ 追问 → 存入会话。

**故事 2（老师建组与公开资料）**
> 作为一名老师，我希望建一个学习小组并上传课件，只把整理好的部分对学生公开，以便学生按需学习、而我保留备课草稿。

流程：创建 team → 上传资料 → 对精选资料开 `shared` → 学生加入后可见公开资料，草稿仅自己可见。

**故事 3（学生私有资料 + 加入小组）**
> 作为一名学生，我希望把课堂笔记存进自己的私有 team，同时加入老师的小组看公开资料，以便我的 AI 学伴既能讲课本也能讲我自己的笔记。

流程：资料上传至私人 team → 加入老师 team → AI 答疑时同时检索多个知识库，引用标明来源。

**故事 4（规划）**
> 作为一名备考者，我希望告诉学伴「两周后考教资」，它能生成每日计划。

流程：进入 AI 学伴 → 说目标与期限 → Planner 生成计划 → 同步学习中心 → 每日打卡。

**故事 5（测评反馈）**
> 作为一名大学生，学完一章后我希望做一套测验，以便知道哪里薄弱。

流程：学习中心 → 生成测评 → 作答 → Evaluator 批改 → 薄弱点报告 + 推荐资料。

## 7. 非功能性需求

| 维度 | 要求 |
|------|------|
| 性能 | 资料列表 < 300ms；AI 首字延迟 < 1.5s（流式） |
| 可用 | 核心路径可用性 ≥ 99.5% |
| 安全 | JWT 鉴权、RBAC、team 可见性、行级隔离、传输加密 |
| 兼容 | 主流浏览器；移动端响应式 |
| 成本 | Agent 调用按订阅档位限流（free 每日额度 / pro 放宽），token 按用户记账，防滥用 |

---

## 8. 商业模式（钱从哪来）

| 模式 | 说明 | 备注 |
|------|------|------|
| 个人订阅 | Free / Pro 月费，Pro 解锁无限问答、高级规划；档位落到 Agent 限流额度与 `token_usage` 记账 | 主营收 |
| 增值功能 | 测评深度分析、导出计划 | 提升 ARPU |
| B 端 / 机构 | 班级版、学校版按席位付费（基于 team 体系天然适配） | 规模化关键 |
| 内容合作 | 与教辅/课程方分成 | 后期 |

> 产品顾问视角「钱」：先验证个人付费意愿（Pro 订阅），再用机构版（多 team）放大。MVP 阶段不建议一上来做重 B 端。

## 9. 关键指标（分发与增长）

| 类别 | 指标 | 说明 |
|------|------|------|
| 活跃 | DAU / WAU | 学习频次 |
| 参与 | 人均学习时长、问答次数、team 活跃数 | 粘性 |
| 留存 | 次日/7 日/30 日留存 | 学伴是否真的被需要 |
| 转化 | 注册→Pro 付费率 | 商业验证 |
| 效果 | 测评正确率提升、计划完成率 | 产品价值证据 |

> 产品顾问视角「分发」：早期靠老师带班冷启动（一个老师 = 一个 team = 一批学生），比纯广告更高效。

---

## 10. 竞品参考（简述）

- 搜题类 App：强答疑、弱体系与陪伴。
- 网课平台：强内容、弱个性化与即时性。
- 通用 ChatGPT：强通用、弱「基于我的资料/我的班级」与学习闭环。
- **本产品差异点**：团队知识库 + 多 Agent 协作 + 学习闭环 + 进度记忆 + 师生共建资料。

## 11. 里程碑 / 路线图

| 阶段 | 时间（估） | 目标 |
|------|-----------|------|
| M0 原型 | 第 1–4 周 | 三端打通：资料上传→自动解析→RAG 答疑→对话历史（含文件存储/解析状态机） |
| M1 MVP | 第 5–8 周 | F1–F6 + F2.5 + 解析基座上线（含三类角色、班级码审批、team 知识库），小范围内测 |
| M2 增强 | 第 9–12 周 | 学习计划 + 测评 + 看板（F7–F10） |
| M3 增长 | 第 13 周+ | 订阅上线、机构版（多 team）探索（F11+） |

## 12. 风险与对策

| 风险 | 对策 |
|------|------|
| 回答幻觉 | RAG + 引用溯源 + 低置信度提示 |
| Agent 成本高 | 限流、缓存、模型路由 |
| 冷启动无资料 | 预置公共库示例资料、支持导入 |
| 权限越界 | team 可见性 + `shared` 强制过滤 + 检索层谓词由后端下发 + 审计日志 |
| 内容风险 | K12 输入/输出内容安全网关 |
| 留存低 | 计划/测评闭环制造回访理由 |

---

## 13. 范围边界（本期不做）

- 不替代学校正式教学与考试。
- 不提供学历/证书背书。
- MVP 不含移动原生 App（先做响应式 Web）。
- 跨用户资料共享（除 team 机制外）本期不做。
